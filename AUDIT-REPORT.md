# Co.AI Monorepo — Forensic Static Audit

**Scope:** `aof-web/` (Next.js 14 frontend + API), `tmap-v2/` (Express TMAP backend), `coagentix-cli/` (CLI). Static/structural only — source, architecture, UI consistency, dead code, doc-vs-code drift. Deliberately does not duplicate `qa-loop/`'s 57-phase runtime QA or `validation-engine/`'s scorecard harness, both of which already cover live/browser-driven behavior.
**Method:** full reverse-reference dependency mapping per package, `typecheck`/`lint` runs, and manual grep/read verification of every claim below. One live smoke test was also run (`aof-web` dev server + Playwright driver across `/`, `/chat`, `/code`, `/projects`, `/settings`, `/login` plus a live chat round-trip) — referenced where relevant, not a substitute for `qa-loop`.
**Not modified:** no source files were changed to produce this report.

## 1. Executive Summary

- **Critical — conversation history silently loses data at the UI layer.** `aof-web/src/app/api/conversations/[id]/messages/route.ts` exports only `POST`; no `GET` handler exists anywhere, and the `messages` Supabase table is never `SELECT`ed. Any conversation loaded from a different device, or evicted from the 20-message localStorage cache, renders as an empty "Welcome to Co.AI" thread with **no error shown** — the data is safe in Supabase, but the user cannot see it. §4.1
- **High — the CLI's default entry point skips its own safety pipeline.** `coagentix-cli`'s interactive REPL (`coai` with no arguments) applies AI-written file changes via only preview→snapshot→apply, bypassing the security gate, reliability scoring, patch validation, checkpoint/auto-rollback-on-build-failure, and ownership recording that every single-shot command (`fix`, `generate`, `refactor`, …) gets. Its own snapshot mechanism's rollback function is dead code, so REPL-applied changes aren't actually recoverable despite printing a snapshot ID. §2.3, §4.4
- **High — a well-designed error-code registry is used by 1 of 50 API routes.** `aof-web/src/lib/errors/api-error.ts` + `error-codes.ts` (15 typed codes) is documented as mandatory ("All API routes must use these helpers") but only `keys/route.ts` actually does. 47 routes return ad-hoc untyped `{error: string}` shapes with inconsistent status codes; `admin/analytics/route.ts` doesn't even check Supabase errors — a failed query silently becomes `0`/`[]` with zero logging. §4.5
- **High — CoCode's core IDE navigation is entirely unusable by keyboard.** The file tree, tab strip, and collapsible sections in `file-explorer.tsx`, `diff-viewer.tsx`, `monaco-editor.tsx`, and others are built on raw clickable `<div>`s with no `role`, `tabIndex`, or keydown handling — you cannot open a file or switch tabs without a mouse anywhere in the audited sample. §3.4
- **High — a documented architectural claim is false.** `aof-web/CLAUDE.md` states the CLI's Tool Execution Engine is "where tools actually run" — but `coagentix-cli/src/tools/` (6 files, a faithful hand-synced duplicate of `tmap-v2`'s tool registry) has zero call sites anywhere in the package; it compiles but nothing ever invokes it. §2.3
- **Medium, root-cause — no shared panel shell.** 29 of 70 CoCode panels hand-copy an identical header class-string; ~10 more diverge structurally; no `PanelShell`/`PanelHeader` component exists to enforce or fix this in one place. This is the single root cause behind most of the UI-consistency findings in §3. §5.A
- **Positive, worth stating plainly:** all three packages' `typecheck` is fully clean; `aof-web`'s `lint` is fully clean; the chat pipeline has real zod validation at its trust boundary and an honestly-implemented streaming invariant; the deploy pipeline explicitly refuses to fake unconfigured targets; `tmap-v2`'s apparent duplicate-looking module pairs (dual orchestrators, dual budget layers, dual memory/quality-gate modules) all turned out to be intentional, documented layering with zero actual logic duplication on inspection.
- **Doc drift is real but asymmetric:** `CLAUDE.md` (the governance doc, updated per-phase) checked out accurate in every package; `README.md`/`ARCHITECTURE.md` files (onboarding-facing, not phase-gated) are substantially stale in both `aof-web` and `tmap-v2`. §5.F

## 4. Workflow / Data-Flow Integrity Findings

### 4.1 Conversation history load — Critical, confirmed broken
`aof-web/src/app/api/conversations/[id]/messages/route.ts` exports **only `POST`** (lines 30-72) — no `GET`. Confirmed via full-repo grep that `.from("messages")` is called exactly once anywhere in `aof-web/src` (the `POST` handler's `.upsert()`) — the table is never `SELECT`ed. `chat-store.ts`'s `loadRemoteConversations()` (lines 174-201) hardcodes `messages: []` for every remote conversation (line 187) and never attempts a message fetch. The sidebar (`sidebar.tsx:189-195`) opens a conversation via `selectConversation(id)` only — no lazy-load-on-open hook exists anywhere in the codebase. **Impact:** any conversation from a different browser/device, or one whose messages fell outside the 20-message localStorage cap (`chat-store.ts`'s `partialize`, line ~493), opens to an empty "Welcome to Co.AI" state with the sidebar still showing a correct title/timestamp — the conversation *looks* intact until clicked, and no error is surfaced. Data is safe server-side; this is a pure UI-layer regression.

### 4.2 Error-handling consistency — High
Sampled/grepped all 50 route files. **1 of 50 (2%)** — `keys/route.ts` — actually uses the documented `api-error.ts` + `error-codes.ts` typed pattern end-to-end. Two more (`chat/route.ts`, `refactor/route.ts`) import the *separate* legacy `@/lib/errors` system (`AofProviderError`, 13 `AOF_ERROR_0xx` codes) instead — and `refactor/route.ts:5` imports `classifyProviderError` but **never calls it** (dead import; its actual error paths at lines 23/26/31/35 are raw untyped JSON). The remaining 47 routes return ad-hoc `{error: "<short-string>"}` shapes with route-local, inconsistent status-code choices (e.g. `conversations/route.ts` alone uses at least 5 distinct micro-vocabularies: `"invalid-workspace"`, `"delete-failed"`, `"messages-required"`, `"save-failed"`). `admin/analytics/route.ts` (lines 63-99) doesn't check Supabase call errors at all — a failed query silently resolves to `0`/`[]` in the API response with no logging, meaning a broken analytics query looks identical to "genuinely zero" from the client's perspective. Even `chat/route.ts` — the best-covered route — has an internal inconsistency: its "message required" 400 response (line 360) is a third, untyped shape inside a file that already imports a typed error system.

### 4.3 CoCode build/deploy pipeline — no Critical/High findings
`build-panel.tsx` → `code-store.ts` → live `tmap-v2` SSE endpoints, with an honest `/api/chat` fallback rather than fabricated output. `deployment-panel.tsx`/`lib/cocode/deployment.ts` explicitly limit themselves to what's real (ZIP export, GitHub Pages) and show "Not configured" for Vercel/Netlify/Railway/Cloudflare rather than faking a deploy — this is one of the more defensively-written surfaces in the app. One forward-compat note: `api/github/route.ts:13-16`'s `getToken()` calls `cookies()` synchronously without `await` — correct on the app's current Next.js 14.2.35, but this exact pattern is what breaks silently on a Next 15 upgrade (where `cookies()` becomes async); no compile-time guard exists against that drift.

### 4.4 coagentix-cli interactive REPL bypasses the safety pipeline — High
Every single-shot command (`fix`, `generate`, `refactor`, `docs`, `debate`, default `coai <task>`) routes through `cli.ts`'s `applyWithConfirm` (lines 88-172): security-gate scan → reliability scoring (with a reject gate) → patch generation/validation (path-traversal/protected-path checks) → checkpoint → apply → build validation with **auto-rollback on failure** → ownership recording. The interactive REPL (`interactive.ts:runTask`, lines 170-250) — which is the CLI's *default* mode, i.e. what runs when a user just types `coai` — applies the exact same category of AI-proposed file changes via only `previewAndConfirm` → `createSnapshot` → `applyChanges`, skipping every one of those six protections. Compounding this, `createSnapshot`'s paired `rollbackSnapshot` function (§2, `files.ts`) has zero call sites — so even the one safety net the REPL does create (a `.coai-snapshots/` entry, whose ID is printed to the user) cannot actually be rolled back through `coai checkpoint --rollback` (which only reads `.coai-checkpoints/`, an entirely separate directory) or any other exposed command.

### 4.5 Orphan API routes — 17 confirmed (aof-web)
See §2's table row for the full list. All 17 are fully implemented (real zod validation, auth, several with genuine Supabase persistence) and carry systematic "Phase 46-70" comment tags — the same numbering convention visible on the (all UI-wired) "Phase 71-100" CoCode panels. No caller was found in `aof-web/src`, `tmap-v2/src`, or `coagentix-cli/src`. This is flagged as **needs-human-confirmation** rather than a hard defect (§7) — systematically-tagged, fully-built backend work is more consistent with "shipped ahead of its UI" or "meant for an external/future caller" than with accidental dead code, but that can't be confirmed by static reading alone.

## 5. Root-Cause Patterns

**A. No shared panel-shell abstraction.** 29-of-70 hand-copied header strings, 4 independently-reimplemented popup/menu components, and the general "each cocode panel is its own bespoke layout" texture (§3) all trace back to one missing extraction: a `PanelShell`/`PanelHeader` (and ideally a shared `usePanelMenu`-style hook wrapping `ui/dropdown-menu.tsx`). Building that once fixes the header drift in ~30-40 files and gives the 4 reimplemented-menu files a natural home to migrate onto.

**B. Backend build cadence runs ahead of frontend wiring, in both directions.** The 17 orphan `aof-web` API routes (fully built server-side, no UI caller) and the orphaned Smart Keyboard subsystem (fully built frontend, never mounted into the app tree, own comments admitting the integration step was skipped) are the same underlying pattern viewed from opposite sides: work gets phase-numbered and shipped as a unit, but the "wire it into the tree/nav/menu" step is sometimes a separate, droppable task that isn't tracked as part of "done."

**C. A correct pattern exists but nothing enforces adoption.** The typed error-code registry (§4.2, 2% adoption), the Tool Execution Engine in `coagentix-cli` (§2, built, documented as canonical, zero callers), and the REPL-vs-single-shot safety gap (§4.4, the hardened path exists but isn't the only path to the same outcome) are all instances of "the right implementation shipped once, in one place, and was never made the *only* way to do the thing." None of these are lint-enforceable by default (no route generator/template, no CLI-wide dispatcher requiring all mutation paths to go through one gate).

**D. No lint gate in `tmap-v2` or `coagentix-cli`.** Both packages rely on `tsc --noEmit` alone. Type-checking cannot catch unused exports, dead imports, or drift — exactly the class of finding (`refactor/route.ts`'s dead `classifyProviderError` import, 10 dead `coagentix-cli` functions) that this audit had to find by manual grep sweep rather than a `no-unused-vars`-class rule catching it automatically pre-merge.

**E. Governance docs are phase-gated; onboarding docs are not.** `CLAUDE.md` in `aof-web` checked out accurate against every specific claim tested (Kanon/Mikros/Ypertatos invariants, `prestream-dispatch.ts`'s dispatch order, prompt-compiler layer order) — it's clearly updated as part of shipping each phase. `README.md`/`ARCHITECTURE.md` in both `aof-web` and `tmap-v2` are substantially stale (missing component/store directories, wrong roadmap status, a broken cross-repo doc link, an install path from a different developer's machine, "Phase 1 MVP" framing on a codebase with a fully-built kernel/recovery/certification layer) — these files simply aren't part of the same update discipline.

## 6. Remediation Roadmap (sequenced)

**Phase 0 — immediate, no dependencies (est. 1-2 days combined)**
1. Add a `GET` handler to `conversations/[id]/messages/route.ts` and wire `chat-store.ts`'s `loadRemoteConversations()` to actually fetch message content. (§4.1 — user-facing data-loss-at-UI-layer, highest priority in this report.)
2. Add error checking + logging to `admin/analytics/route.ts`'s Supabase calls. (§4.2, cheap, closes a silent-failure gap.)

**Phase 1 — foundational, unblocks Phase 3/4 work (est. 1 week)**
3. Extract `PanelShell`/`PanelHeader` from the 29-file convention; migrate the ~10 structural outliers onto it as part of the same change. Do this *before* touching individual panels for anything else. (§5.A)
4. Decide the error-registry question: commit to `api-error.ts`/`error-codes.ts` as the standard (retrofit the highest-traffic routes first: `chat`, `conversations/*`, `admin/*`) or fold it into `errors.ts`'s already-adopted system — don't leave two systems at 2%/70% adoption indefinitely. (§4.2, §5.C)
5. Add ESLint (at minimum `no-unused-vars`/`no-unused-imports`) to `tmap-v2` and `coagentix-cli`. (§5.D — mechanically prevents a chunk of this report's Low findings from recurring.)

**Phase 2 — safety parity (est. 3-5 days)**
6. Bring `coagentix-cli`'s interactive REPL onto `applyWithConfirm`'s pipeline (or explicitly gate/label it as a reduced-safety mode with user consent). (§4.4)
7. Wire `files.ts`'s `rollbackSnapshot` into an exposed command so REPL-applied changes are actually recoverable. (§2, §4.4)

**Phase 3 — accessibility (est. 1-2 weeks, can parallelize with Phase 4)**
8. Convert `file-explorer.tsx`'s directory/file/tab rows to real interactive elements (`role`, `tabIndex`, keydown) first — it blocks the most core navigation. Then `diff-viewer.tsx`, `monaco-editor.tsx`, `testing-agent.tsx`, `simulation-engine-panel.tsx`, `api-studio.tsx`. (§3, §4.4)
9. Sweep `aria-label` onto icon-only buttons across `cocode/` — do this as part of the Phase 1 `PanelShell`/menu migration where possible, since a shared header component can standardize icon-button treatment in one place.
10. Fix the two color-only status indicators (`devops-panel.tsx`, `infrastructure-panel.tsx`).

**Phase 4 — dead-code resolution (est. 3-5 days, mostly deletions)**
11. Product decision + action on: Smart Keyboard subsystem (finish wiring or delete, 6 files), `coagentix-cli/src/tools/` (wire or delete, 6 files), `coai tasks` (wire `launchBackgroundTask` or remove the command).
12. Delete confirmed-dead, no-decision-needed items: `ide-prefs-store.ts`, `lib/cocode/github-api.ts`, the 10 dead `coagentix-cli` functions.
13. Migrate the 4 hand-built popup/menu implementations in `cocode/` onto `ui/dropdown-menu.tsx` — natural to bundle with Phase 1's shell work.
14. Consolidate or scope the 13-file hardcoded console-color pattern into a single named utility.

**Phase 5 — doc hygiene (est. 2-3 days, can run anytime, no code dependencies)**
15. Rewrite or explicitly mark-historical `aof-web/ARCHITECTURE.md` and `tmap-v2/README.md`.
16. Fix `tmap-v2/README.md`'s broken `AOF_CODE_TDD.md` link and stale install path.
17. Document the `errors.ts`/`errors/` coexistence (or resolve it per Phase 1 item 4).
18. Fix `mascot/palette.ts`'s stale palette-rationale comment.

**Phase 6 — investigate / lower priority (no fixed timeline)**
